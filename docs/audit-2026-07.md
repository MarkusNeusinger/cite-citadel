# Audit 2026-07 — where citadel stands, and what is still missing

An extensive point-in-time audit (2026-07-20, v0.4.0 + unreleased `refresh`): an internal
code-level review, a survey of comparable implementations (LLM-maintained knowledge bases and
memory systems), and a scan of the surrounding tooling landscape (MCP servers, local retrieval,
agentic ingest) — synthesized into one ranked gap backlog.

**TL;DR.** Citadel's quadrant — *verified, cited, human-readable synthesized knowledge with
explicit maintenance lifecycles* — was nearly empty until Karpathy's viral "LLM Knowledge Bases"
post (2026-04) spawned a wave of direct implementations, the largest at ~15k stars within months.
Even so, no surveyed system combines source-grounded ingestion, machine-enforced per-fact locator
citations, deterministic offline verification, and staged all-or-nothing promotion — that stack
remains citadel's moat. The exposure is on the **access** side, not the knowledge side: ranked
retrieval (the FTS5 swap already scoped in `store_core.search()`), a conversational-capture
bridge, and cost observability are the three gaps most likely to dominate any side-by-side
comparison — and each has a design that closes it without compromising the provenance guarantees.

## 1. Internal audit

### 1.1 What the repo already tracks as open

- **The search upgrade is the one explicit "Future" marker in the code** —
  `store_core.py:185` scopes the swap ("replace this body with SQLite FTS5 bm25 — signature +
  MCP surface stay identical") behind the single `search()` seam. Nothing has been started.
- **Unfilled model-benchmark cells** — the README's Opus/Sonnet/Haiku matrix: leuchtfeuer,
  pemberley, and clockwork have never been benchmarked on any tier; beverages/flurfunk lack
  Opus/Sonnet runs. Each filled cell is a single non-deterministic ingest (self-flagged).
- **`refresh` is unreleased** — in CHANGELOG `[Unreleased]` while `__version__` is 0.4.0; a
  release PR is a pending act.
- **The verify-corpus "two optimization lanes" backlog is ephemeral** — every grading miss is
  routed into a creation lane (rules/prompts) or a retrieval lane (search), but the lane findings
  live only in per-run grade reports; nothing is persisted in the repo, so misses that didn't
  become fixes are lost.
- **Documented, accepted trade-offs** (missing by design, not oversights): chunked-source
  all-or-nothing (a failure at segment N discards N−1 segments' work), no self-update command,
  `curate`/`refresh`/`view` CLI-only, no offline PDF/image reader, best-effort OLE salvage,
  best-effort wiki-git, provider-side rate limiting.

### 1.2 Code-level findings

1. **Contradiction "detection" is marker-counting, not detection.** `lint.py` counts existing
   `> [!CONTRADICTION]` callouts, and curate's `contradiction` reason plans work only for
   already-flagged callouts. An unflagged semantic contradiction between two pages is invisible
   to every offline layer — surfacing it depends entirely on the ingest agent noticing at write
   time (exactly the miss the Haiku benchmark shows).
2. **Locators into PDF/Office/image sources are never offline-verified.** `lint.check_locators`
   skips all non-text extensions (`lint.py:103` `_NON_TEXT_EXTS`), and `wiki_raw` returns "read
   it directly" for PDFs/images (`rawsource.py:39`). For that whole source class, "every locator
   verifiable" silently degrades to "trust the agent".
3. **Token/cost usage is discarded.** `llm.py:630-643` parses the claude `--output-format json`
   envelope only for `is_error`; its usage/cost fields are dropped. `refresh` is *marketed* as
   budget management, yet the `--limit` unit is sessions, not tokens or cost, and nothing records
   what a session actually cost.
4. **`CITADEL_INGEST_MODEL` only applies to the `claude` backend** — silently inert on
   copilot/gemini, and `doctor` does not warn about the combination.
5. **Config is frozen at import and process-global.** A long-lived `citadel serve` never sees
   `.env` edits; one process = one workspace; and ingest's staging redirect mutates
   `config.WIKI_DIR` + `os.environ` in place — safe under the strictly-serial design, a hard
   blocker for any future parallel ingest or multi-workspace server.
6. **MCP read tools reload the entire wiki from disk on every call** (`store.load()` per call,
   no cache) — see § 1.3 for the scale impact.
7. **`extract.py`/`extract_ole.py` swallow everything** — `extract_text` returns `""` on ANY
   error, so an encrypted or corrupt Office file is indistinguishable from a truly text-free one;
   both land as `unreadable` with the same generic detail.
8. **Two divergent search implementations** — the Python `search()` is OR-semantics token
   overlap; the viewer's JS search is AND across fields with `tag:`/`type:` operators. The same
   query can hit in the viewer and miss over MCP/CLI.
9. **`log.md` is unbounded and re-read on every append** — O(log size) per source ingested,
   forever.

### 1.3 Retrieval-layer assessment

Today's `search()` is purely lexical bag-of-words presence: a one-suffix-layer stemmer, field
weights (title 3.0 → body 1.0), smoothed per-query IDF, a flat substring bonus. **No term
frequency, no length normalization, no BM25 saturation, no phrase/proximity, no fuzzy match, no
embeddings, no persisted index** — and every query re-walks and re-parses the whole wiki from
disk. Findability at showcase scale (tens of pages) is graded and good; at 1000+ pages both
latency (O(corpus disk I/O) per MCP call) and ranking will degrade first. The intended fix (FTS5
BM25 behind the unchanged seam) is scoped in-code but unstarted.

### 1.4 Test-coverage map (thin spots only)

Coverage is broad (44 test files, ~1:1 with modules, codecov patch gate 80%). The relative gaps:

| Surface | State |
|---|---|
| `extract_ole.py` (222-line hand-rolled CFBF binary parser) | 2 offline tests; real coverage is the kontor corpus, which needs a live LLM run |
| `viewer/app.js` (~2k lines: search, graph layout, popovers, keyboard nav) | zero automated tests — Python build/bundle only |
| `lint.py` | no dedicated test file; functionally covered but scattered across four files |
| `store_core.neighbors_text`, `append_log`, `catalogs.py`, `linkgraph.py` | indirect/thin only |

### 1.5 Operational gaps

- **Per-source staging cost is O(wiki size) × pending sources** — each source (and curate
  cluster) full-copies and re-hashes the live wiki; a 100-source ingest of a 1000-page wiki
  copies the wiki ~200×. Nothing (hardlinks, dirty-set copies) mitigates it yet.
- **Strictly serial** — no intra-run parallelism knob (blocked by finding 5), while the per-source
  staging copy is already exactly the isolation primitive parallel ingest would need.
- **No watch mode / scheduler** — `refresh --min-age-days` is designed for external cron, but
  scheduling is entirely the user's problem; no file-watcher exists.
- **`wiki_ingest` over MCP blocks the stdio loop** for the whole multi-source run, with no
  progress or async-job surface.
- **No privacy story beyond local-model recipes** — wiki, raw, manifest, and unredacted session
  transcripts (`CITADEL_LLM_LOG_DIR`) are plaintext.
- **Recovery is best-effort** — wikigit is advisory, no restore path, no manifest backup, and a
  chunked source loses all paid-for segments on a late failure.

## 2. Landscape: comparable implementations

Surveyed: the Karpathy-lineage implementations (below), basic-memory, khoj, mem0, Letta
(MemGPT), cognee, Graphiti/Zep, txtai, LlamaIndex, Obsidian ecosystem (Copilot, Smart
Connections, Bases, vault MCP bridges), Logseq, Notion AI, Tana, Capacities, Mem, Anthropic
memory (app, API memory tool, Claude Code auto-memory), ChatGPT/Gemini memory, and the MCP
memory-server registry landscape. *(The Karpathy-lineage, basic-memory/khoj, and
assistant-memory/MCP-server lanes were verified against primary sources on 2026-07-20; the
mem0/Letta/cognee/Graphiti/commercial-PKM lane is synthesized from project knowledge current
through early 2026 — spot-check fast-moving details there.)*

**Key finding: the pattern now has direct competitors.** Karpathy's "LLM Knowledge Bases" post
(2026-04-03, [tweet](https://x.com/karpathy/status/2039805659525644595)) and his follow-up
[idea gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) — three layers
(raw → wiki → schema), three operations (ingest/query/lint), "compile knowledge once and keep it
current" instead of RAG — went viral and spawned implementations within months:

- **[nashsu/llm_wiki](https://github.com/nashsu/llm_wiki)** (~15k stars): Tauri desktop app;
  ingests PDF/Office/EPUB/images/web clips (Chrome extension); per-page `sources:[]` frontmatter
  + numbered citations; tokenized search + optional LanceDB vectors + 4-signal graph expansion;
  **human review queue**, community detection, cascade deletion, lint tools; MCP server;
  Obsidian-compatible.
- **[lucasastorian/llmwiki](https://github.com/lucasastorian/llmwiki)** (~1.4k stars): MCP-first
  (full-text search + citation-graph tools), Chrome clipper, footnote citations, Next.js viewer;
  distinctive: **scheduled nightly Claude Routines** autonomously fold new sources in.
- **[Astro-Han/karpathy-llm-wiki](https://github.com/Astro-Han/karpathy-llm-wiki)**: an
  agent-skills package (raw→wiki, ingest/query/lint, cited answers) — the "share the idea, not
  the app" lane. Plus a ["v2" design gist](https://gist.github.com/rohitg00/2067ab416f7bbe447c1977edaaa681e2)
  adding decay, supersession links, and hybrid retrieval past ~200 pages, and several smaller
  entrants (MindBase, Eva, okf-gem, …).

The rest of the landscape still splits into (a) memory layers extracting facts from
*conversations* (mem0, Letta, ChatGPT/Claude memory), (b) RAG/search over *unmodified* notes
(khoj, Obsidian plugins, txtai), and (c) markdown KBs written incrementally *during chats*
(basic-memory — which, note, ships **hybrid FTS5+FastEmbed-vector retrieval by default**).
Maintenance-over-time machinery exists only in the Karpathy lineage — and none of the new
entrants has machine-enforced citation validation, a locator grammar, transactional staged
promotion, deletion-propagation, budgeted refresh, or graded evaluation corpora. Practitioner
consensus from the gist thread names **staleness/drift as the #1 failure mode** — exactly what
citadel's reconcile+refresh already address.

### 2.1 Where citadel is ahead

1. **Per-fact cited provenance with locators, machine-enforced** — no surveyed system stores
   knowledge where every fact carries a verifiable citation (line ranges / § headings) into an
   immutable source, with the `[^sN]`/`[^llmN]` raw-vs-model distinction and a validation gate.
   The new Karpathy-lineage tools cite at page/footnote granularity but nothing *checks* the
   citations; Graphiti's edge→episode links lack locators and human readability; LlamaIndex
   proves locator citations only at *answer* time — citadel does it at *storage* time.
2. **Deterministic offline verification** (check/lint with CI exit codes) — memory quality
   elsewhere is vibes + benchmarks.
3. **Four explicit maintenance lifecycles** (reconcile, deletion-propagation, curate, budgeted
   refresh) — mem0's ADD/UPDATE/DELETE loop, Letta's sleep-time agents, and Graphiti's temporal
   fact-invalidation are each a partial analogue; none combine all four, none staged with rollback.
4. **Transactional ingestion** (staging, diff-by-hash, promote-once, run locks) — competitors
   mutate a DB in place.
5. **No API key / BYO agent CLI; zero-infra portability.**
6. **Human-readable, git-diffable knowledge as the artifact** — shared only with basic-memory and
   the Obsidian world, neither of which synthesizes.
7. **Adversarial-injection resistance as a graded corpus** — no surveyed competitor publishes
   prompt-injection hardening tests for its ingestion path, despite it being a live attack class.

### 2.2 Common in the landscape, absent in citadel (ranked)

1. **Semantic / hybrid retrieval** — near-universal (khoj, mem0, cognee, Graphiti, txtai,
   basic-memory, Copilot, Smart Connections, mcp-memory-service, nashsu/llm_wiki's optional
   vectors). Citadel is lexical-only. Partially mitigated
   by curated aliases/cross-links and the query-iterating agent consumer, but oblique-phrasing
   recall is the single biggest measurable gap (and a verify-corpus lane).
2. **Automatic capture from conversations** — mem0, Letta, basic-memory, ChatGPT/Claude memory
   all grow the KB as a side effect of chatting; citadel only eats files dropped in `raw/`. A
   conversation-transcript→`raw/` bridge or a `wiki_remember`-style attributed capture tool would
   close it without breaking provenance.
3. **Chat-with-your-KB end-user UI** — citadel's "any MCP client is the UI" is defensible in
   2026, but the offline viewer is browse-only.
4. **Connector-based ingestion** (Notion/Slack/email/web-clipper, watch folders) — even a generic
   "fetcher writes into raw/" convention would slot into the existing lifecycle.
5. **Sync / mobile / multi-device** — markdown+git makes DIY sync easy; it is undocumented as a
   recipe.
6. **Voice / meeting capture** — entirely absent (see § 3.3: transcript ingest is the natural
   next format).
7. **Formal temporal validity model** — Graphiti's bi-temporal `valid_at`/`invalid_at` preserves
   "what was believed when"; citadel handles supersession in prose + Open Points, not queryably.
8. **Hosted / multi-user / permissions** — a real landscape norm, likely out of scope by
   philosophy.
9. **Always-on related-content surfacing** (Smart Connections' pane) vs. on-demand `neighbors`.
10. **Decay/forgetting policies** — refresh re-verifies but nothing ages out; arguably a feature.

## 3. Tooling landscape

### 3.1 MCP surface

Modern peers (cyanheads' Obsidian server, Basic Memory, the reference memory server, Notion's
hosted server) ship capabilities citadel's stdio-tools-only server lacks: **prompts** (packaged
workflows exposed as slash-command-like entries), **resources** with `listChanged`/subscribe
notifications (pages as addressable URIs), **Streamable HTTP** transport (+ auth) for
claude.ai/web/multi-device reach, elicitation, and experimental long-running tasks (the natural
fit for a non-blocking `wiki_ingest`). Citadel already does two modern things most peers skip:
tool annotations and `initialize.instructions`. Every serious memory product now leads with an
MCP server — the MCP-first design is squarely on trend, and the 12-tool read surface is richer
than most. Peers also all have a write surface; if citadel ever adds one, it should go through
the `[^llmN]` provenance lane (e.g. an append-only observation/open-point tool the next curate
pass folds in properly), never raw file edits.

### 3.2 Retrieval without a vector DB

The 2026 evidence *validates* citadel's lexical bet: Anthropic replaced vector search in Claude
Code with grep-style agentic search in 2025, most coding agents followed, and agentic keyword
search measures near-RAG faithfulness with zero vector store. Since citadel's consumer is always
an agent that iterates queries, embeddings matter less than they look — but **ranked** search
still beats substring matching on first-shot hit rate. The pragmatic ladder:

1. **SQLite FTS5** — zero new dependencies (stdlib `sqlite3`), BM25 ranking, phrase/prefix/NEAR,
   per-column weighting (boost title/aliases over body). At personal-wiki scale, an in-memory
   FTS5 table built at `load()` keeps "the wiki IS the database" literally true — no on-disk
   index, no staleness. Slots behind the existing `search()` seam exactly as scoped.
2. **A lightweight reranker** (FlashRank-class: no torch, ~4 MB, sub-20 ms for 50 candidates)
   over the top lexical hits — quality bump with no index at all.
3. **Hybrid as an opt-in extra at most** — `pip install cite-citadel[semantic]` → fastembed +
   sqlite-vec + reciprocal-rank fusion, only if the verify-corpus oblique-query lane proves
   lexical+aliases insufficient. Never a required dep. Avoid rank-bm25 (strictly worse than
   FTS5) and BM25S (static-corpus assumption fights the incremental lifecycle).

### 3.3 Agentic ingest patterns

- **Cost/metrics surface**: claude's `--output-format json` returns `total_cost_usd` + usage;
  gemini has `--session-summary`. Parsing these per session and stamping cost into the manifest
  (surfaced in `status`/reports) is near-zero code and answers "what does my wiki cost to
  maintain" — directly serving refresh/curate's budget framing (finding 1.2.3).
- **Hermetic sessions**: claude's `--bare` skips user hooks/CLAUDE.md/MCP discovery — today a
  user's `~/.claude` config leaks into citadel's agent sessions; `--bare` + explicit flags is the
  reproducibility fix.
- **Resumability**: session-ID capture (`--resume`) could make chunked large-source ingest
  resumable at segment N instead of the current retry-from-segment-1 trade-off.
- **Parallelism**: per-source staging is already the isolation primitive; a bounded `--jobs N`
  (promotes serialized under the run lock) is the credible throughput win — blocked today by the
  process-global config mutation (finding 1.2.5).
- **Audio/video is the largest un-ingestable content class left**, and fits the no-SDK philosophy
  perfectly: a `CITADEL_AUDIO_SUPPORT` knob shelling out to a whisper-CLI binary on PATH (exactly
  like the agent-CLI seam), caching the transcript as the text the session reads, plus a
  `formats/transcripts.md` brief (timestamps as locators, speaker attribution per the flurfunk
  rules).
- **PDF text-layer pre-pass**: optional pypdf (pure Python, BSD) extraction would give real
  `lines A-B` locators and correct chunker sizing while keeping agent-native reading for figures;
  avoid AGPL/model-heavy parsers as deps. Citadel's hand-the-PDF-to-the-agent design is otherwise
  the 2026-idiomatic "VLM does the OCR" pattern.
